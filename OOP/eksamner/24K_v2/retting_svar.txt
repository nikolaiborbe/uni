## Del 2

Dette er en faglig skjønnsmessig retting. Poengene under er omtrentlige, siden faktisk sensur kan vekte formuleringer annerledes.

### 13. Testing

**Omtrentlig poeng: 1.5/5**

Du forklarer systemtesting delvis riktig, men enhetstesting er feil forklart. Enhetstesting handler ikke primært om feil typer, men om testing av små isolerte enheter, for eksempel én funksjon eller én klasse. Regresjonstesting mangler.

Korrigert svar:

> Enhetstesting tester små deler av programmet isolert, for eksempel en funksjon eller klasse. Systemtesting tester hele programmet samlet. Regresjonstesting er testing etter endringer for å sjekke at tidligere fungerende funksjonalitet ikke har blitt ødelagt.

### 14. `auto`

**Omtrentlig poeng: 4/5**

Hovedideen er riktig: typen utledes fra uttrykket på høyresiden. Eksempelet med lange typer er relevant. Formuleringen om "lvalue-en" er litt upresis.

Korrigert svar:

> Når en variabel deklareres med `auto`, utleder kompilatoren typen fra initialiseringsuttrykket. Det er nyttig når typen er lang eller tungvint å skrive, for eksempel iteratorer eller komplekse templatetyper.

### 15. `scale`

**Omtrentlig poeng: 4/5**

Du ser at begge argumenter gis som referanser, og at `factor` burde vært gitt på en annen måte. Det beste svaret er at `number` bør være referanse fordi den endres, mens `factor` bør være verdi eller eventuelt `const double&`, men for en `double` er verdi best.

Korrigert svar:

> Begge argumentene gis som referanser. Det er riktig for `number`, siden funksjonen skal endre argumentet. `factor` bør heller sendes som verdi, siden den ikke endres og `double` er billig å kopiere.

### 16. Referanser

**Omtrentlig poeng: 4/5**

Svaret er godt. Du nevner både endring av argumenter og å unngå kopiering. Du kunne også nevnt flere returverdier og `const`-referanser for skrivebeskyttet tilgang.

Korrigert svar:

> Referanser er nyttige fordi funksjoner kan endre argumenter direkte, man kan unngå dyre kopier av store objekter, man kan bruke `const`-referanser for skrivebeskyttet tilgang, og man kan lage funksjoner som effektivt returnerer flere verdier via referanseparametere.

### 17. `std::map` og `push_back`

**Omtrentlig poeng: 3/5**

Du er inne på at `std::map` ikke er en sekvensbeholder, men forklaringen med red-black tree og at det ikke finnes et "siste element" er litt upresis. Et `map` har en siste iterator i sortert nøkkelrekkefølge, men `push_back` gir ikke mening fordi elementene sorteres etter nøkkel, ikke innsettingsrekkefølge.

Korrigert svar:

> `std::map` lagrer elementer sortert etter nøkkel. Derfor gir det ikke mening å legge noe "bakerst" slik `push_back` gjør for sekvensbeholdere som `std::vector`. Nye elementer plasseres etter nøkkelen sin.

### 18. Lagring og pekere

**Omtrentlig poeng: 2/5**

Du har noen riktige deler, men flere viktige feil. `print` er global og ligger i statisk lager, ikke på stacken. Selve `std::vector`-objektet `a` ligger på stacken, men dataene vektoren eier ligger typisk på heapen. Pekeren `ptr` ligger på stacken, mens `int`-objektet den peker på ligger på heapen.

Korrigert svar:

> `print` ligger i statisk lager. `a` ligger som vektorobjekt på stacken, mens elementdataene ligger typisk på heapen. `b`, `diffN`, `n` og lokale parameter-/kontrollvariabler ligger på stacken. `ptr` ligger på stacken, mens objektet laget med `new int(1)` ligger på heapen fram til `delete ptr`.

### 19. Klasser

**Omtrentlig poeng: 0/10**

Du svarte "jeg vet ikke" på begge delene.

Korrigert svar:

> a) Koden kompilerer ikke fordi `B` fortsatt er abstrakt. Klassen `B` arver den rent virtuelle funksjonen `foo()` fra `A`, men implementerer den ikke. Derfor kan man ikke lage `B b(3, 4);`.
>
> b) For å løse problemet uten å endre `main`, må `B` implementere `foo()`.

Eksempel:

```cpp
class B : public A {
public:
    B(int x_, int y_) : A(x_, y_) {}

    void foo() override {
        std::cout << "foo" << std::endl;
    }

    void bar() const override {
        std::cout << "bar" << std::endl;
    }
};
```

### 20. Header-filer

**Omtrentlig poeng: 5/5**

Riktig. `f1.h` inkluderes flere ganger, slik at `get_pi()` defineres flere ganger. `#pragma once` øverst i `f1.h` løser det. Det er også god praksis å bruke det i `f2.h`.

### 21. Templates

**Omtrentlig poeng: 7/10**

Idéen er riktig, men koden har syntaksfeil: manglende semikolon etter `return x`, `return y` og etter klassedefinisjonen. Forklaringen i b) er riktig.

Korrigert svar:

```cpp
template <typename T>
class Point {
    T x = 0;
    T y = 0;

public:
    T getX() {
        return x;
    }

    T getY() {
        return y;
    }
};
```

Fordelen er at samme klasse kan brukes med ulike datatyper, for eksempel `Point<int>` og `Point<double>`.

### 22. `>>` og `getline()`

**Omtrentlig poeng: 2/5**

Du forklarer `getline()` ganske riktig, men `>>` er feil forklart. `operator>>` stopper ikke bare ved newline; den leser formatert input og stopper normalt ved whitespace eller når input ikke passer typen.

Korrigert svar:

> `operator>>` leser formatert input til en bestemt type og stopper typisk ved whitespace eller når input ikke passer typen. `getline()` leser en hel linje fra strømmen fram til newline eller en annen gitt delimiter.
